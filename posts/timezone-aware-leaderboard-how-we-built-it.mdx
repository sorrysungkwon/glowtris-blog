Then I hopped on a plane, landed in New York, and suddenly I was back in "yesterday."

Did my daily streak break? Nope. Not at all.

I play Duolingo every single day. But one day, I flew to New York and realized my leaderboard ranking hadn't changed. My streak was perfectly intact! My timezone had completely flipped, yet everything was fine.

I ran into the exact same dilemma when building the "Daily Leaderboard" for Glowtris.

I needed a leaderboard that resets every midnight... **But whose midnight are we talking about?** Seoul's midnight? New York's midnight? UTC midnight?

The answer is **"My midnight."** Regardless of where I am in the world.

## The Disaster of UTC Reset

The most one-dimensional approach is wiping the DB entirely at UTC midnight.

But if you do that, players in Seoul will see the leaderboard vanish right at 9 AM when they are actively playing. Players in New York will log in after work at 7 PM, only to have the leaderboard wipe right in front of their eyes. It completely kills the thrill of a competitive daily leaderboard.

Should we manage 38 separate leaderboards for every timezone in the world? Server costs would explode, and you'd have the bizarre situation where Seoul players can't compete with their friends in New York.

## The Solution: Splitting into Date Buckets

Instead of "wiping" the leaderboard, I changed the architecture to "create a new one per date."

```text
glowtris-daily-2026-06-04  ← Yesterday
glowtris-daily-2026-06-05  ← Today (Seoul)
glowtris-daily-2026-06-06  ← Today (New York)
```

When a user connects, their device sends its **local timezone date** to the server.

```javascript
const d = new Date()
const localDate = `${d.getFullYear()}-${String(d.getMonth()+1).padStart(2,'0')}-${String(d.getDate()).padStart(2,'0')}`

// User: "I'm in Korea, and it's June 5th right now!"
fetch(`/api/leaderboard?date=${localDate}`)
```

The server simply and blindly serves the leaderboard for that specific date bucket. When midnight passes and the device date changes? The app naturally fetches the next day's bucket. There's absolutely no need to delete old data.

## "Wait, couldn't users just fake their date and cheat?"

Absolutely. Trusting the client 100% is suicidal. A user could manipulate the script to send yesterday's date and illegally post high scores on past leaderboards.

So, I added a single line of validation logic on the server.

```javascript
function safeClientDate(raw) {
  const utc = new Date().toISOString().slice(0, 10)
  if (!raw || !/^\d{4}-\d{2}-\d{2}$/.test(raw)) return utc
  
  // Cut off any date that is more than 24 hours away from current UTC
  return Math.abs(new Date(raw) - new Date(utc)) <= 86400000 ? raw : utc
}
```

The largest timezone offset on Earth is +14 hours (Kiribati). This means no legitimate user can ever send a local date that differs from current UTC by more than 24 hours. Valid global users pass through smoothly, while malicious manipulations get instantly blocked.

## ‍️ Setting the Server TTL to 52 Hours

We can't let these daily Redis buckets pile up forever, so they need a TTL (Time-To-Live). I initially set it to 26 hours, which almost ended in a bloodbath.

The time difference between the start of the day in the earliest timezone (UTC+14) and the end of the day in the latest timezone (UTC-12) is a massive **50 hours**. So, to be safe, I set the TTL to 52 hours.

## Conclusion and Checklist

The server shouldn't arrogantly force its UTC time on users. It should simply respect the user's local device time.

Because of this, when I play in Seoul and land in New York, my phone tells the server it's "yesterday," seamlessly attaching me back to yesterday's leaderboard bucket to keep my streak alive.

If you're building a global service, just remember these 3 things:
1. Don't wipe leaderboards; split them into date buckets.
2. Accept the client's local date, but only within a ±24 hour UTC window.
3. Set your Redis expiration to at least 50 hours.
