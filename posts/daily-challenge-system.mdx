I shipped Glowtris with a daily leaderboard that was supposed to "reset" at midnight in your local timezone. Like Duolingo's weekly leagues — you finish your day, the board feels fresh, you check back tomorrow. That's the hook.

But I almost made it stupid.

I nearly shipped a leaderboard where Seoul and San Francisco share the same "day" — both looking at `glowtris-daily-2026-06-05`. The moment you get on a plane from Seoul to SF, you'd go from rank 1 to rank 47 without playing a single game, because you're suddenly competing against a different 24 hours of players.

Yeah. That would've been embarrassing.

Here's how I almost broke it, and how it actually works now.

## The broken approach I almost shipped

My first instinct: "Let me just add a cron job that wipes the leaderboard at UTC midnight. Done."

I literally wrote this:

```javascript
// ❌ DON'T DO THIS
const wipeLeaderboard = async () => {
  await redis.del('glowtris-daily-leaderboard')
}

// Run at UTC midnight
schedule.scheduleJob('0 0 * * *', wipeLeaderboard)
```

I didn't even think about timezones. I wrote it at 2am, tested it locally, and was ready to push.

Then I thought: "Wait. If this runs at UTC midnight, when does it run in Seoul?"

UTC midnight = 9am Seoul time. 

So you're grinding on the leaderboard in the morning, the board suddenly wipes in the middle of your session, and everyone resets at the exact same moment regardless of where they are. The whole "midnight refresh" feeling is gone.

The other dumb idea: "Run a cron job for every timezone."

That's 38 canonical IANA timezones. You'd need 38 separate leaderboards and 38 separate cron jobs. Technically possible. Practically a nightmare to maintain.

## The actual fix: server keeps time, not the client

Claude pointed me at the real solution: **the server keeps a single global clock. UTC.**

No date-key switching based on local midnight. The server just does:

```javascript
const today = new Date().toISOString().slice(0, 10)  // UTC
const key = `glowtris-daily-${today}`
```

That's it. Every player in every timezone writes to the same bucket on the same calendar day (UTC). When UTC midnight hits, the server flips to the next day. Done.

No client date validation. No timezone database. No "is this date legitimate?" checks. Just: what's the UTC date right now?

Here's what happened to me:

I'm in Seoul, grinding on the leaderboard on June 5. It's 23:50 (Seoul time). Midnight hits (00:00 Seoul), which is 15:00 UTC. Server checks the date: still June 5 UTC. Same bucket. But then actual UTC June 5 ticks to June 6, and the server flips to the next key.

But here's the thing: I was playing **before** UTC midnight, so my score is still on June 5's board.

I get on a plane. Land in San Francisco. It's 6:50 AM on June 5 (SF time). But on the server, it's still June 5 UTC (well, it was). I open Glowtris. The board is exactly where I left it. My streak is unbroken. Because the server doesn't care what time it is in San Francisco — it only cares what time it is in UTC.

That moment when I realized the leaderboard didn't break? I was shocked at how simple it was.

One global clock. Everybody sees the same day. Your local time zone is irrelevant.

## Why this is so much simpler

The server-side approach has zero security problems. Nobody sends a date. The server just checks its own clock.

The old system I *almost* shipped (client sends local date) would've required:

```javascript
function safeClientDate(raw) {
  const utc = new Date().toISOString().slice(0, 10)
  if (!raw || !/^\d{4}-\d{2}-\d{2}$/.test(raw)) return utc
  // reject anything more than 1 day away from UTC today
  return Math.abs(new Date(raw) - new Date(utc)) <= 86400000 ? raw : utc
}
```

Why? Because someone could send `date=2026-01-01` and try to manipulate the leaderboard.

But with the server doing it? No validation needed. The server is the source of truth. Full stop.

## TTL: when do old boards expire?

Keys need to expire. I set it to 52 hours.

Why 52 and not 24?

The server keeps UTC. UTC June 5 is a single 24-hour window. But because the Earth has timezones, players in UTC+14 are living June 6 while the server is still calling it June 5 UTC. 

So when the server says "June 5 is over, let's move to June 6," there are still players in UTC+14 who are literally living June 5 locally. They'll submit scores to `glowtris-daily-2026-06-05` for another 14 hours.

52 hours gives everyone in all timezones time to finish their "day" before the old bucket expires. Nobody loses a score because the server moved too fast.

---

So yeah. One server clock. Boom. Done.

You're grinding in Seoul at 11:50pm. Midnight hits. Board flips. You get on a plane. Land in SF 12 hours later (your local time)—but on the server it's been like 2 hours. Board's still there. Streak intact. You're still in the same "day" as everyone else on the planet.

That's it. That's the whole trick. The server doesn't care what time it is in San Francisco. It doesn't care what time it is *anywhere*. It just knows what time *it* is.
