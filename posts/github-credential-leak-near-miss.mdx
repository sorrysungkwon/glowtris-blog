During late-night military study time, I pushed my security keys to a public GitHub repository.

I didn't receive a single warning email.

So, blissfully unaware, I cheerfully clicked deploy on Vercel.

## 😱 What Happened

Usually, when keys are leaked, you expect GitGuardian or AWS to send a warning email saying "Your security keys have been exposed!" But... **I didn't receive any warning emails.**

What I had pushed to the public repo were my **security credentials** and my **Moshi webhook token**. Since I got no emails, I had absolutely no idea they were leaked and peacefully continued developing.

## 🔥 Why Is This So Scary? (The Recent Tving Security Incident)

Remember the massive **Tving data breach affecting 13 million users** that occurred in early June 2026?

The starting point of that catastrophic incident wasn't an advanced hacking technique, but **an AWS access key exposed on an open-source platform (GitHub)**. Hackers stole the key leaked during development, accessed their internal database, and extracted CI, DI, and encrypted passwords for 13 million users. This single mistake has led to an ongoing massive class-action lawsuit.

What if someone's bot had scraped the tokens and keys from my public repo? They could have spun up hundreds of crypto miners on my account, leaving me with a bill for tens of thousands of dollars overnight, or completely wiped my small server.

## 🤔 Why Didn't the Warning Email Come?

Thinking about it still gives me chills, but why didn't the system detect it? Here is my assumption:

- **Moshi webhook tokens are not standard formats.** Secret Scanning from GitGuardian or GitHub prioritizes **known patterns** like AWS keys starting with `AKIA` or GitHub PATs. Custom service tokens might not be pattern-recognized, easily bypassing the scanners.
- **I hadn't integrated any security scanner into my repo at the time.**

Conclusion: **"No warning means it's safe" is a dangerous myth. Warning systems are not perfect.**

## 🏃‍♂️ My Action History (Trigger Warning: Terrifying)

Initially, completely unaware of the leak, I proudly ran the following commands to upload and deploy:
```bash
gh repo create tetris --public --push
vercel deploy
```
That's the most terrifying part.

However, I later realized that my **Moshi webhook token (`9W8b67...`) was hardcoded and publicly exposed** in `robot.md` and `CLAUDE.md`. Panicking, I immediately executed the following remediation steps. If you experience a key exposure, please follow these steps quickly.

### 1. Local Backup & Isolation
I backed up all sensitive credentials, including the exposed token and Admin password, into a server-only `CREDENTIALS.md` file locally.

### 2. Add to `.gitignore`
To prevent this backup file from ever causing another disaster, I immediately added it to `.gitignore`.
```text
# .gitignore
.env
.env.local
.env.*

# Sensitive backup documents
CREDENTIALS.md
```

### 3. Instantly Revoke & Reissue Token
I instantly invalidated the exposed Moshi token and generated a brand new one (`G5aOFC...`). Once it's in the GitHub commit history, just deleting the file isn't safe. **You must absolutely revoke the token itself.**

### 4. Pair & Test New Token
I tested the new webhook token via `curl` to ensure it worked, and securely re-paired it to the system.
```bash
moshi-hook pair --token G5aOFC...
```

## 💡 Lessons Learned & Practical Checklist

You might think "I'd never do that," until you do and it destroys everything in one shot. I recommend doing these 2 things right now.

### 1. Setup Global .gitignore (Set it once, use it forever)
```bash
echo -e ".env\n.env.local\n.env.*.local\nCREDENTIALS.md" >> ~/.gitignore_global
git config --global core.excludesfile ~/.gitignore_global
```

### 2. Integrate GitGuardian for Free (Takes 5 mins)
It feels like locking the stable door after the horse has bolted, but right after this incident, I newly integrated [GitGuardian](https://www.gitguardian.com/) into my GitHub repository to automatically detect and block any future key exposures by bots.

Today, I've resolved to make a habit of checking `git status` three times before pushing code.
