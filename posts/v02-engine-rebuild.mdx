v0.2 was me rewriting the entire game engine in a week. The game looks exactly the same. Nobody would know I did it unless I told them.

But on Reddit, someone said it felt "crisper." That's when I knew the old engine was actually broken.

## The Reddit complaints that changed everything

People on r/vibecoding started leaving comments:

> "The game feels... off. Like input lag or something. idk"

> "Why does holding right sometimes feel slow, sometimes feel instant?"

> "Rotation doesn't feel snappy. Compared to Tetr.io it's kinda mushy."

I wasn't trying to build a "feels good" game. I was trying to build *a* game. But Tetris is muscle memory. Your fingers know what "right" feels like, and I'd built "wrong."

So I opened up the original code:

```javascript
function tick() {
  updateGame()
  render()
  setTimeout(tick, 16) // "60fps"
}
tick()
```

This was the problem. It seemed fine, but:

**Problem 1: setTimeout lies about timing.** Browsers clamp it to ~1-4ms minimum, and under load it drifts badly. Your "16ms tick" might fire at 18ms, 22ms, or 45ms. For Tetris — where gravity, DAS timing, and lock delay are measured in single-digit milliseconds — that drift destroys consistency.

**Problem 2: It's chained to the wrong thing.** Logic and rendering happen on the same `setTimeout` cadence. On a 144Hz monitor, you're still getting 60fps — the monitor's extra refresh rate is just ignored. All those extra frames wasted.

I tried a quick fix: switch to `requestAnimationFrame`:

```javascript
function loop(now) {
  updateGame()
  render()
  requestAnimationFrame(loop)
}
requestAnimationFrame(loop)
```

This made rendering silky smooth on 144Hz monitors. Good.

But now game logic ran at 144Hz on gaming rigs and 60Hz on laptops. Gravity was 2.4x faster on high refresh rate monitors. The whole game felt different depending on your screen. That's worse than before.

## The fix: fixed timestep with an accumulator

The solution is called the **fixed timestep with accumulator pattern**. It's boring but it works: 

- Run game logic at a **fixed, consistent rate** (mine: 1000Hz)
- Run rendering at **whatever the monitor can do** (60Hz, 144Hz, whatever)
- Decouple them completely

```javascript
const TICK_MS = 1 // 1000 logic updates per second

let accumulator = 0
let lastTime = performance.now()

function loop(now) {
  const dt = Math.min(now - lastTime, 100) // cap to avoid spiral of death
  lastTime = now
  accumulator += dt

  while (accumulator >= TICK_MS) {
    gameTick()  // runs at exactly 1000Hz regardless of monitor
    accumulator -= TICK_MS
  }

  render()  // runs at whatever the display refresh rate is
  requestAnimationFrame(loop)
}
```

Why 1000Hz? Because Tetris DAS (delayed auto-shift) timings are measured in **1ms increments** in competitive play. At 60Hz, the minimum expressible delay is 16.67ms — invisible rounding errors that add up. At 1000Hz, it's exactly 1ms.

Does it cost CPU? Negligibly. My logic loop is simple.

The `Math.min(dt, 100)` cap matters: if your tab backgrounded for 10 seconds, you don't simulate 10,000 ticks at once. You skip to "now" and resume.

## Input handling: the invisible problem

Here's the problem Reddit was actually complaining about: input events fire whenever the OS feels like it, not synchronized with your game ticks.

Say a `keydown` fires at t=1.3ms into a 1ms tick. If you process it immediately, you've applied it 0.3ms "early" relative to the game clock. Two inputs arriving in the same tick get processed in random order depending on when the OS fired the events.

The fix is a **timestamp queue**:

```javascript
const inputQueue = []

document.addEventListener('keydown', e => {
  inputQueue.push({ key: e.key, t: performance.now() })
})

function gameTick(tickStart) {
  // Only process inputs that arrived before this tick started
  while (inputQueue.length && inputQueue[0].t <= tickStart) {
    processInput(inputQueue.shift())
  }
  // ... rest of game logic
}
```

Now inputs are processed in the tick when they **actually happened**, in the right order, at the right time.

Result: input lag dropped from ~4ms average (one setTimeout frame) to ~0.3ms. You can't consciously feel the difference, but your fingers do. That's why people said the new version felt "crisper."

They weren't making it up. The old engine actually had 4ms of invisible delay baked in.

## Why this matters (even if nobody notices)

The game looks the same. But 4ms of invisible latency is gone.

Your fingers know the difference. Reddit knew it. I didn't—not until I read the complaints.

I could've shipped it as-is. It would've been "fine." But fine isn't good enough when someone's hands know better. So I spent a week rebuilding the invisible parts, the parts nobody can see in a screenshot, the parts that only matter when you're actually playing.

That's infrastructure work. Nobody notices it until it's missing.

I notice it now.
