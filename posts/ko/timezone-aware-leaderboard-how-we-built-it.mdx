글로우트리스에 '데일리 리더보드'를 만들면서 똑같은 딜레마에 빠졌습니다.

매일 자정에 리셋되는 리더보드를 만들어야 하는데... **도대체 누구 기준의 자정이냐는 겁니다.** 서울 자정? 뉴욕 자정? UTC 자정?

정답은 **"내 기준의 자정"**입니다. 내가 전 세계 어디에 있든 상관없이 말이죠.

## UTC 기준 리셋의 대참사

가장 1차원적인 방법은 매일 UTC 자정에 DB를 싹 지우는 겁니다.

하지만 이렇게 하면 서울 유저는 아침 9시 한창 게임할 때 리더보드가 증발해버립니다. 뉴욕 유저는 퇴근하고 저녁 7시에 게임을 켰는데 갑자기 리더보드가 리셋됩니다. "매일 순위가 바뀌는 쫄깃한 리더보드" 느낌이 다 죽어버리는 거죠.

그렇다고 전 세계 38개 시간대별로 리더보드를 따로 운영한다? 서버 비용은 폭발하고 서울 리더보드와 뉴욕 리더보드가 갈라지는 기괴한 상황이 생깁니다.

## 해결책: 날짜별 버킷(Bucket) 쪼개기

리더보드를 "지우는" 대신, "날짜별로 새로 만드는" 구조로 바꿨습니다.

```text
glowtris-daily-2026-06-04  ← 어제
glowtris-daily-2026-06-05  ← 서울 기준 오늘
glowtris-daily-2026-06-06  ← 뉴욕 기준 오늘
```

유저가 접속할 때, 유저 기기의 **로컬 시간대 날짜**를 서버에 던져줍니다.

```javascript
const d = new Date()
const localDate = `${d.getFullYear()}-${String(d.getMonth()+1).padStart(2,'0')}-${String(d.getDate()).padStart(2,'0')}`

// 유저: "나 한국인데 지금 6월 5일이야!"
fetch(`/api/leaderboard?date=${localDate}`)
```

서버는 무식하게 그 날짜로 된 리더보드를 보여줄 뿐입니다. 자정이 지나서 기기 날짜가 바뀌면? 자연스럽게 다음 날 버킷을 쳐다보게 됩니다. 데이터를 굳이 지울 필요도 없습니다.

## "그럼 날짜 조작하면 다 털리는 거 아닌가요?"

맞습니다. 클라이언트를 100% 믿는 건 자살 행위죠. 유저가 스크립트를 조작해서 어제 날짜를 보내면 예전 리더보드에 점수를 올릴 수 있습니다.

그래서 서버 단에 1줄짜리 검증 로직을 넣었습니다.

```javascript
function safeClientDate(raw) {
  const utc = new Date().toISOString().slice(0, 10)
  if (!raw || !/^\d{4}-\d{2}-\d{2}$/.test(raw)) return utc
  
  // UTC 오늘과 24시간 이상 차이나는 미래/과거 날짜는 컷.
  return Math.abs(new Date(raw) - new Date(utc)) <= 86400000 ? raw : utc
}
```

지구상에서 UTC와 가장 많이 차이나는 곳이 +14시간(키리바시 공화국)입니다. 즉, 어떤 유저든 현재 UTC와 하루(24시간) 이상 차이 나는 날짜를 보낼 수 없습니다. 합법적인 글로벌 유저는 모두 통과되고, 악의적인 조작은 칼같이 차단됩니다.

## ‍️ 서버 TTL은 52시간으로 세팅

날짜별 Redis 버킷은 영원히 쌓이면 안 되니 TTL(만료 시간)을 걸어줍니다. 처음엔 26시간으로 세팅했다가 피를 볼 뻔했습니다.

가장 빠른 시간대(UTC+14)의 하루 시작점과, 가장 느린 시간대(UTC-12)의 하루 끝점 차이는 무려 **50시간**에 달합니다. 그래서 여유를 두고 TTL을 52시간으로 세팅했습니다.

## 결론 및 체크리스트

서버는 오만하게 자신의 UTC 시간을 유저에게 강요하지 않습니다. 유저 기기의 현지 시간을 존중할 뿐이죠.

그래서 제가 서울에서 게임을 하다가 비행기를 타고 뉴욕에 내려도, 폰이 "어제로 돌아갔다"고 알려주기 때문에 자연스럽게 전날 리더보드 버킷에 붙어서 스트릭을 이어갈 수 있는 겁니다.

글로벌 서비스를 만들고 계신다면 이 3가지만 기억하세요.
1. 리더보드는 지우지 말고 날짜별로 쪼갤 것
2. 클라이언트의 로컬 시간을 받되, UTC 기준 ±24시간까지만 허용할 것
3. Redis 만료 기간은 최소 50시간 이상으로 잡을 것
