v0.2는 일주일간 전체 게임 엔진을 다시 짠 작업입니다. 게임은 똑같아 보입니다. 제가 말하지 않는 한, 아무도 알 수 없습니다.

하지만 Reddit에서 누군가 "더 반응적이더라"고 했을 때, 나는 원래 엔진이 실제로 망가졌다는 걸 깨달았습니다.

## Reddit이 지적한 것들

r/vibecoding에서 댓글들이 올라오기 시작했습니다:

> "게임이... 이상해. 뭔가 입력 지연 같은? 모르겠지만"

> "오른쪽 누를 때 왜 어떨 땐 느리고 어떨 땐 순간적으로 움직이는 거야?"

> "회전이 안 날카로워. Tetr.io와 비교하면 좀 뭔가 푸석푸석한 느낌이야."

저는 "느낌 좋은" 게임을 만들려고 한 게 아니었습니다. 그냥 게임을 만들려고 했었어요. 하지만 테트리스는 근육 기억입니다. 손가락이 "맞는" 느낌을 알고 있고, 저는 "틀린" 걸 만들었거든요.

그래서 원래 코드를 열었습니다:

```javascript
function tick() {
  updateGame()
  render()
  setTimeout(tick, 16) // "60fps"
}
tick()
```

이게 문제였습니다. 보기는 좋았지만:

**문제 1: setTimeout은 거짓말을 합니다.** 브라우저가 최소 1~4ms로 제한하고, 부하 상태면 더 심하게 어긋납니다. "16ms 틱"이 18ms, 22ms, 45ms가 될 수 있습니다. 테트리스에서 중력·DAS·락 딜레이는 **1ms 단위**입니다. 그 편차는 일관성을 파괴합니다.

**문제 2: 잘못된 것에 연결되어 있습니다.** 로직과 렌더링이 같은 `setTimeout` 주기로 움직입니다. 144Hz 모니터에서도 60fps 렌더링만 얻습니다. 나머지 프레임은 낭비.

## 고정 타임스텝: 누산기 패턴

표준적인 방법이 있습니다. 이름은 거창하지만 원리는 단순합니다:

- 게임 로직은 **일정한 속도로** 실행 (저는 1000Hz)
- 렌더링은 **모니터가 지원하는 속도로** 실행 (60Hz든 144Hz든)
- 완전히 분리

```javascript
const TICK_MS = 1 // 초당 1000번

let accumulator = 0
let lastTime = performance.now()

function loop(now) {
  const dt = Math.min(now - lastTime, 100) // 과부하 방지
  lastTime = now
  accumulator += dt

  while (accumulator >= TICK_MS) {
    gameTick()  // 항상 1000Hz
    accumulator -= TICK_MS
  }

  render()  // 모니터 속도에 맞춰
  requestAnimationFrame(loop)
}
```

왜 1000Hz일까요? 테트리스 DAS(방향 반복)는 경쟁 플레이에서 **1ms 단위**로 측정되거든요. 60Hz면 16.67ms 단위—눈에는 안 띄지만 손가락은 느껴집니다. 1000Hz면 정확히 1ms.

CPU 비용? 무시할 수준입니다.

`Math.min(dt, 100)` 부분이 중요한데, 탭이 배경에 있다가 돌아왔을 때 10초치를 한 번에 처리하려는 걸 막으려고입니다.

## 입력 처리: 눈에 띄지 않는 문제

고정 타임스텝에는 함정이 하나 있습니다. 입력은 OS가 주는 대로 들어옵니다. 틱과 무관하게.

키가 1ms 틱의 1.3ms 지점에서 눌렸다고 하면, 그걸 바로 처리하면 0.3ms "일찍" 처리하는 셈입니다. 같은 틱에 들어온 두 입력이 실제 순서와 다르게 처리될 수 있습니다.

**타임스탬프 큐**로 고칩니다:

```javascript
const inputQueue = []

document.addEventListener('keydown', e => {
  inputQueue.push({ key: e.key, t: performance.now() })
})

function gameTick(tickStart) {
  while (inputQueue.length && inputQueue[0].t <= tickStart) {
    processInput(inputQueue.shift())
  }
  // ...
}
```

이제 입력은 **실제로 일어난 시점의 틱**에서, **정확한 순서대로** 처리됩니다.

결과: 입력 지연이 ~4ms (setTimeout 한 프레임)에서 ~0.3ms로 뚝 떨어집니다. 의식적으로는 못 느끼지만 손가락은 안다. 그래서 Reddit 사람들이 "더 반응적이더라"고 한 거고.

그들이 틀린 게 아니었어요. 원래 엔진에 4ms의 보이지 않는 지연이 있었던 거예요.

## 왜 이런 일을 했나

게임은 똑같아 보입니다. 점수도 같고, 속도도 같습니다. 하지만 누군가 "더 반응적이더라"고 했고, 그 사람이 맞았어요.

경쟁 테트리스 클라이언트는 다 이렇게 만들어집니다. 우아해서가 아니라 필수라서.

게임을 100판 할 때쯤이면 손가락이 기억해요. 4ms 지연을.

그냥 놔뒀으면 게임은 돌았을 거고, 나중에 누군가 "뭔가 이상한데?" 하면 그때 다시 써야 했을 걸요.

대신 일주일을 써서 기반부터 다시 만들었으니, 다시 손댈 일은 없습니다.

게임은 똑같고, 느낌은 다릅니다. 때론 그게 다입니다.
